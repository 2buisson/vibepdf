# Decision log — spec-remove-mvvm

## 2026-06-27 — Create

**Input.** In-chat one-liner: "remove CommunityToolkit.Mvvm and re-develop everything without MVVM architecture." Sparse intent; brownfield refactor. No source file — `sources: []`.

**Pivotal decision asked.** "Without MVVM" forks into materially different specs depending on what happens to the ~45 headless `MainViewModel` tests. Presented three targets: (1) code-behind + plain testable core, (2) full code-behind collapse, (3) drop toolkit but keep MVVM. **User chose (2) full code-behind collapse** — delete `MainViewModel`, fold everything into `MainWindow.xaml.cs`; the headless tests are deleted or rewritten as UI tests. This is recorded as a hard Constraint (the MVVM pattern goes, not just the package) and rules out option 3.

**Distillation mode.** Express, grounded in a full read of the codebase (csproj, `MainViewModel`, `PdfFileItem`, `MainWindow.xaml`/`.xaml.cs`, `App.xaml.cs`, `MergeResultEventArgs`, `BoolToVisibilityConverter`, and `MainViewModelTests`). 3 capabilities, 1 spec-authored companion (`brownfield.md`).

**Self-validate — Pass 1 (Coherence).** Rules 1–6, 8. Each capability has intent + success (1). Intents state WHAT; the HOW (construct→replacement map) is pushed to `brownfield.md` (2). Constraints bend design — "no MVVM pattern" rules out the hand-roll path; "keep DI/services" rules out gutting the service layer (3). Non-goals explicit (4). Success signal is a concrete F5 walkthrough + green service tests (5). Capability IDs CAP-1..3, stable (6). Prose kept lean (8). Two genuine gaps could not be resolved without a human and became Open Questions: test disposition, and how list-item status refreshes without `[ObservableProperty]`. Three inferred calls (scope = MVVM only, `ObservableCollection` may stay, converter is dead code) became Assumptions.

**Self-validate — Pass 2 (Preservation).** The input is a one-line intent; its single load-bearing claim ("remove the toolkit AND the MVVM architecture") is carried by CAP-1 + CAP-2 and the Constraints. The load-bearing *codebase* facts a behavior-preserving refactor must honor (toolkit construct inventory, concurrency invariants, behavior edge-cases, file-level blast radius) are preserved in `brownfield.md` so they are not silently lost in the rewrite. No wrapper-only content to drop.

**Verdict.** Coherent and preservation-complete for the chosen target (full code-behind collapse). Two open questions are live and should be resolved before implementation, as both change scope: the test-suite disposition (regression safety net) and the list-item update mechanism (truest-to-no-MVVM imperative refresh vs. lighter hand-rolled INPC on the model).

## 2026-06-27 — Update: open questions resolved

Both open questions answered by the user and folded into SPEC.md (as Constraints) and `brownfield.md`; the Open Questions section is now removed.

- **Test disposition → delete the VM tests.** `MainViewModelTests` is deleted outright, not ported. Post-refactor regression net = `PdfValidationServiceTests` + `PdfSharpMergeServiceTests` + the manual F5 parity walkthrough. Accepted trade-off: no headless coverage of merge-gating, selection-shift, or validation-timeout logic survives.
- **List-item updates → imperative row refresh.** `PdfFileItem` becomes a plain class with no `INotifyPropertyChanged`; on validation completion the code-behind updates the row via `FileListView.ContainerFromItem(item)`. Fully binding-free (truest to "no MVVM"). Noted caveat: `ContainerFromItem` returns null for virtualized/unrealized rows — the refresh must no-op safely.

**Verdict (updated).** No open questions remain. Spec is implementation-ready for the full code-behind collapse. Residual risk is concentrated in the concurrency invariants (`brownfield.md` §Concurrency behaviors to preserve) and the imperative row-refresh fragility, both now explicitly flagged for the implementer.
